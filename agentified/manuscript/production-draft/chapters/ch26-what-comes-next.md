## Chapter 26: What Comes Next

The office lights had timed out once and come back on when Sam moved. An hour ago the building had decided Sam did not count as motion. That was fine. The quiet was the point. The six beats of the day had run their course. Friday closed the same way every week now, with a seventh beat only Sam kept.

Two documents sat open on the second monitor. One was called system_next.md. The other was called sam_next.md. Sam had started the ritual months back and had not skipped a Friday since. The rule was simple. Before going home, write what the system needs to know next. Then write what you need to do next. Two different jobs.

system_next.md was full of small, load-bearing things. The support triage agent needed a memory rule for refund thresholds by region. The AutoResearch route needed a stricter gate before pulling from any source published in the last two days. The Foundry rubric needed a new row for agents that touched billing. None of these would ship a headline. All of them would make the system harder to embarrass.

sam_next.md was different. It had three items.

Stop reviewing every approval in the support pod. Trust the pod lead to escalate. Read the weekly digest instead.

Move the standing agent-review meeting from Wednesday morning to Wednesday afternoon. Mornings belong to writing instructions.

Talk to Priya about whether the memory promotion queue should be a rotation instead of a role.

Sam sat back. Six months ago, sam_next.md would have been a list of things to build. Now it was a list of things to instruct, protect, or let go of. Something had quietly changed and Sam wanted to name it before going home.

The change was simple. The work was no longer producing output. The work was governing a system that produced output. Sam had stopped being the person who ships. Sam had become the person who decides what the system is allowed to ship, what it must remember, what it must escalate, and what it must never touch alone.

The Move Fast to Discover section of the "Accelerated" course kit still held. The version for this year needed a small edit. Build the smallest accountable system you can, and improve it in public. The words that mattered were accountable and in public. Anyone could stack agents. The advantage lived in the receipts, the gates, the memory rules, the review rhythm, and the willingness to show the whole thing to a colleague without flinching. [A2]

Sam pulled up a blank note and wrote a name for what had been sitting under the last year of work. Accountability Stack. Five layers. Each one a question the system had to answer before it was allowed to act.

Route. Who or what is doing this work, and how did it get here?

Log. What did the agent see, decide, and produce, in a form a human can read later?

Gate. Where does a human sign before this becomes real?

Memory. What is the system allowed to remember from this, and who promoted it?

Review. When does someone look at all of this together and change the rules?

Sam wrote them top to bottom, then flipped the order and wrote them bottom to top. Both orders taught something. Top to bottom was how a task moved through the system. Bottom to top was how the system got better. Review changed memory. Memory changed gates. Gates changed logs. Logs changed routes. Skip review, and the whole stack calcified into a machine that felt fast and was quietly wrong.

The failure Sam had watched in other teams was almost always the same. They had routes and logs. Sometimes they had gates. Almost none had memory rules. Almost none had review. So their agents got smarter at doing the wrong thing, or drifted into whatever the last prompt-tuner had preferred on Tuesday. No one could say why. No one could roll it back. When something broke, the team argued about the model instead of the stack.

The model was almost never the problem. The stack was the problem.

Sam thought about Priya's pod. A few weeks back, a support agent had drafted a refund reply that would have committed the company to a policy it did not have. The draft went to a human. The human caught it. The reviewer flagged the mismatch. The memory promotion queue rejected the reply as precedent. The route was updated so refund replies above a threshold went to a senior. The next week, the same category of near-miss dropped toward zero. No one wrote a postmortem about the model. They wrote a change to the stack.

That was accountability as a durable advantage. A weekly rhythm, not a slogan.

Sam opened the Portfolio dashboard and pulled up the Friday review view. Five panels. Approvals this week. Recoveries. Memory proposals. Route changes. Evaluation gaps. Every panel had a number and a short list. Every list had a name next to each item, because a change without an owner was a rumor.

Approvals were up. That usually meant either the team was busier or the gates were tighter. The Recoveries panel would say which. Recoveries were down, which meant the gates were doing their job. Memory proposals had a small backlog. Fine. Priya's rotation idea would clear it. Route changes were modest and specific. Evaluation gaps were the panel Sam read last, because that panel told the truth. Two agents were being used for work no one had written an evaluation for. That was the week's real assignment.

Sam added a line to system_next.md. Write evaluations for the two ungated agents before Wednesday. Route them through Foundry. Do not let them keep running on vibes.

Then Sam wrote the exercise that had been forming all week, the one that would go in the training track for new pod leads. Pick one agent. Write what happens when it is wrong. Not in the abstract. Write the exact sentence that appears in the log. Write who sees it first. Write the gate that catches it. Write the memory rule that prevents the next one. Write the review that would notice if the rule stopped working. If you cannot write those five sentences, the agent is not part of the system yet. It is a demo running in production.

One more line before closing the laptop. A question for the pod leads to sit with over the weekend. If your best agent disappeared tonight, what would you lose. The output, or the system around it.

For most teams the honest answer was the output, because most teams had no system around it. The whole point of the last year had been to make the honest answer, for this team, the system. Agents would come and go. Models would be replaced. The route, the log, the gate, the memory, and the review would still be there on Monday, ready to accept a better agent without flinching.

Sam saved both files, closed the laptop, and stood. The building noticed the motion and turned the lights back on for a moment, then let them fade again as Sam walked out.

The note that would go to the pod leads on Monday had one line underlined. Sam had been trying to write it for a year, and it had finally come out short enough to fit on a card.

Stop trying to make agents magical. Build systems that make them accountable.

Chapter 25 of "Accelerated," "The System That Learns," asked teams to build learning systems. "Agentified" asks them to build systems that can be trusted with work. The discipline after this one will not arrive with a new model. It will arrive the morning the stack holds under a load no one planned for, and someone on the pod notices, and writes the next rule. The route, the log, the gate, the memory, and the review are the work now. Everything after is a variation on the same practice, done in public, one Friday at a time. [A1]
